## 一、超级 Agent 集群方案审计

### 1.1 判定：方向正确，方案应换——集群已经存在，不必重建

| 原方案组件 | 审计结论 |
|---|---|
| Claude 1M 做 Orchestrator | ✅ 方向对——但**这就是我本身**。无需用 Python 重新发明我 |
| 自建 asyncio Agent 池 + Celery/Kafka/K8s | ❌ 重复造轮子：我原生具备子代理（Agent）、确定性编排（Workflow，含 pipeline/对抗验证/预算控制）、后台任务、定时远程代理。自建版本更脆弱、更贵、违反"脚手架两周能近似的不做" |
| "CausalAgent = 提示词让 Claude 做 do-calculus" | ❌ **理论性错误**：这正是我们论证过的 L1 鹦鹉问题——因果推理必须由确定性引擎（我们已有的 `causal/`）执行，LLM 只做转译与编排。提示词包装的"因果 Agent"会把幻觉重新请回来 |
| 1M 上下文当数据库（"800K-1M 预留存储"） | ❌ 上下文是工作记忆不是存储。正确模式：**git 仓库为共享真相，登记册为协调协议，每个子代理拿干净上下文** |
| Agent 自我进化（自己写代码扩展自己） | ⚠️ 允许，但必须过同一道门：测试全绿 + 证据分级 + 冻结资产纪律。无人例外，包括 Agent 写的代码 |
| Neo4j/Milvus/Postgres/Prometheus 全家桶 | ❌ 当前规模下是企业级 cosplay。NetworkX>Neo4j、SQLite/JSON>Postgres、文件+git>Kafka——直到数据量证明需要升级 |
| 并行专职化（生成/审查/测试分离） | ✅ 保留——与凭证"生成/校验分离"同构，写入工作流 |

### 1.2 原生超级集群（零新基建，立即可用）

```
Jay（资源·价值·对外） ←→ CC（第一负责人 = Orchestrator 本体）
                              │
        ┌─────────────────────┼──────────────────────┐
        ▼                     ▼                      ▼
   Workflow 编排          Agent 子代理            定时/后台代理
   (确定性 pipeline,      (探索/实现/审查/        (cron 例行审计,
    对抗验证, 并行扇出)     验证, 隔离worktree)      长跑实验值守)

        └────────── git 仓库 = 共享记忆 ──────────────┘
                    登记册 = 协调协议
```

## 二、下一代 The One 总蓝图（M0→M3）

> 主线即论纲 MVP 理论形态：**校准弃权 + 可识别性门控的因果问答 + 证据分级图库**，逐步长成双系统。

### M0 · 已在手（$0，已验证）
因果引擎（do/反事实，F-1 闭）｜A7 弃权判定+δ_min 校准｜凭证+独立校验器｜EG 基准（第一个数据点：有利域 31×）｜T4 跨族门（第一堵墙：OOD 退化 12×）｜两个站桩实验线。

### M1 · 双系统最小闭环（数周，预算 ~$50-200 API 费）🎯 当前主攻
**目标：第一个真正的 System1+System2 混合体跑通，并测出第二个命门数字。**
1. **S1 接入**：DeepSeek API 真实集成（关 E-01，Mock 转真）——LLM 做理解/转译/表达
2. **可识别性门**：实现 ID 判定（后门/前门检查已有雏形）——查询不可识别 → 拒答并指出缺失变量（"因果的'不知道'是可计算的"）
3. **成本敏感路由 v0**：规则版——疑似因果/反事实/高风险 → S2 管道；其余 → S1 直答+置信标注；**宁慢勿错**
4. **证据分级图库 v0**：边携带证据等级（验证/文献/LLM抽取/猜想），低级边不得参与 do-查询
5. **Baseline B 三方对照（预注册）**：裸 LLM vs LLM+脚手架 vs The One——**第二命门数字**，输赢都发表（Q-C8 需 Jack 审样本量）
6. 通用版 Demo 从剧本模拟升级为真实管道驱动

### M2 · 知识与记忆生长（数月，预算 ~$1k-5k）
